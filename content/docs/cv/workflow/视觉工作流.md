# 1. 需求回顾

## 1.1 比赛规则中与任务赛视觉直接相关的约束

任务赛要求机器人在 180 秒内，从 8 个随机分布的物资箱中识别类别并移送到对应归位区，按正确归位得分。任务赛必须为**完全自主控制**，比赛开始后队员不得参与控制。

任务赛场地包含 8 个物资箱、8 个固定存放区、4 个固定归位区、1 条减速带和 1 个智力题显示区。场地尺寸约为 **6000mm × 4000mm**，外围墙内侧为橘黄色。

物资箱共 4 类：**食品、工具、仪器、药品**，每类 2 个；箱体尺寸 **250mm × 250mm × 250mm**，六面贴有**白底黑字黑体**的亚光标识贴。归位区共 4 个，颜色与编号对应为：食品/绿色/0，工具/灰色/1，仪器/蓝色/2，药品/红色/3。

比赛开始时会展示一道四则运算题，机器人需要通过视觉识别题目并给出计算答案，**答案对 4 取模即为高分区编号 0/1/2/3**。若机器人在 **20 秒内**未能识别智力题目并确定高分区，则本轮**没有高分区**。

任务赛中不能在场地内设置任何标识物；物资箱由裁判**随机排放**在存放区；场地和功能区的颜色、尺寸允许有一定误差。

## 1.2 结合你们现有方案的核心功能点

根据你给的输入，你们的视觉系统实际要完成 5 件事：

1. **识别智力题并发布高分区编号**
2. **识别 8 个已知位置上的箱体类别，并建立位置—类别映射**
3. **根据高分区编号，筛选目标箱体位置**
4. **机械臂末端相机对准箱体顶面中心，实现吸取**
5. **搬运后在归位区进行放置**

其中第 2 步里，你们已经明确：
箱体**位置不是靠视觉全场搜索获得**，而是由激光雷达全场定位和已知存放区地图给出，这一点非常关键。它意味着视觉系统的任务重心不是“找箱子”，而是“对固定候选位做分类和抓取定位”。

## 1.3 需求边界

本方案只设计**任务赛视觉技术方案**，不涉及：

- 障碍赛视觉
- 运动控制细节
- 机械臂逆解与底层伺服控制算法实现
- 雷达建图与定位实现细节

但会定义它们与视觉系统之间的接口契约。

# 2. 技术选型

## 2.1 总体选型

### 开发语言

- **Python**：用于视觉算法主流程、模型推理、原型验证、ROS2 节点开发
- **C++（可选）**：用于后期对机械臂视觉伺服、图像处理热点路径做性能优化

### 中间件

- **ROS2 Humble / Jazzy**
  - 原因：你们明确采用 ROS2 发布/订阅；适合模块解耦；便于和导航、机械臂、电控联调

### 深度学习框架

- **PyTorch**
  - 原因：YOLO 训练与部署生态成熟，你们已有经验
- **ONNX Runtime / TensorRT（按平台选）**
  - 原因：比赛现场需要可控延迟；部署时建议转 ONNX，Jetson 再进一步 TensorRT 化

### 图像处理

- **OpenCV**
  - 用于题目区域裁剪、透视矫正、中心偏差计算、轮廓提取、形态学操作、PnP/距离估计辅助

### OCR / 表达式识别

首选方案：

- **专用规则化 OCR 流程 + 轻量 OCR 模型**
- 可选：**PaddleOCR / 自定义字符分类器 / 你们已经验证过的公式识别模型**

不建议把 DeepSeek-OCR 2 直接作为整套任务赛主模型。更合理的定位是：

- 用于**题目识别候选方案**
- 或用于**近距离标签复核**
  而不是负责实时全链路识别

### 检测模型

- **YOLO11n / YOLOv8n / YOLOv8s**
  - 用途一：箱体类别识别
  - 用途二：箱体顶面或顶面标识区域检测
  - 用途三：抓取阶段目标中心检测

选轻量模型是因为你们场景固定、类别少、候选位置已知，没必要上大模型。

## 2.2 数据存储

任务赛不需要传统数据库，建议采用：

- **YAML / JSON 配置文件**
  - 存场地参数、归位区编号映射、相机内参、外参、机械臂抓取高度参数
- **内存态状态管理**
  - 由 ROS2 状态节点维护比赛过程状态
- **CSV / JSON 日志**
  - 记录每次识别结果、抓取结果、耗时，用于复盘

## 2.3 第三方服务

不建议依赖任何在线服务。
原因很简单：比赛现场网络不稳定，自主赛不能把关键链路押在外部服务上。

### 2）业务层（任务决策层）

负责比赛逻辑与状态推进：

- 题目求解
- 高分区判定
- 箱体类别建图
- 抓取任务调度

### 3）数据层（配置与状态层）

## 3.2 总体模块划分

建议拆成以下 ROS2 功能包：

### 包 1：`vision_math_pkg`

负责智力题识别与计算

核心职责：

- 订阅题目相机图像
- 定位题目区域
- OCR / 表达式识别
- 表达式合法性检查
- 求值与 mod 4
- 发布高分区编号和置信度

### 包 2：`vision_box_cls_pkg`

负责箱体类别识别

核心职责：

- 根据存放区固定位置列表，逐个采图/裁图
- 使用 YOLO 对箱体类别分类
- 多帧投票
- 建立 `slot_id -> box_class` 映射
- 筛选高分箱体位置

### 包 3：`vision_grasp_servo_pkg`

负责机械臂抓取阶段的末端视觉伺服

核心职责：

- 订阅机械臂末端相机图像
- 检测箱体顶面区域或顶面标识框
- 计算目标中心与图像中心偏差
- 估计相机到箱体顶面的垂直距离
- 输出机械臂微调指令

### 包 4：`task_orchestrator_pkg`

负责任务状态机与调度

核心职责：

- 控制比赛流程
- 接收高分区编号
- 调度箱体识别
- 生成抓取/放置任务
- 跟踪已抓取数量与剩余目标

### 包 5：`field_config_pkg`

负责地图、参数与标定数据管理

核心职责：

- 存放存放区坐标
- 存放归位区坐标
- 维护相机参数
- 维护抓取高度参数

### 包 6：`robot_interface_pkg`

负责与底盘、机械臂、电控交互

核心职责：

- 将视觉输出转换为标准控制消息
- 与导航包、机械臂控制包进行消息对接

------

## 3.3 推荐系统流程

### 阶段 A：赛前/比赛开始前 20 秒内

1. 机器人面向题目显示区
2. `vision_math_pkg` 识别表达式
3. 计算答案并 `mod 4`
4. 发布 `high_score_zone_id`
5. 同时语音/屏幕播报答案

这一步是比赛规则硬要求。

### 阶段 B：存放区类别扫描

1. 根据固定 8 个存放位坐标，依次观察或在合适视角批量采图
2. `vision_box_cls_pkg` 输出每个 `slot_id` 的箱体类别
3. `task_orchestrator_pkg` 找到类别编号与 `high_score_zone_id` 相同的两个箱体位置
4. 生成抓取顺序

### 阶段 C：抓取

1. 机器狗导航到目标存放位附近
2. 机器狗停止不动
3. 机械臂展开，末端相机垂直向下
4. `vision_grasp_servo_pkg` 检测箱体顶面中心
5. 若未对齐，持续发布平面内偏差
6. 机械臂横向修正直到中心误差低于阈值
7. 估计当前顶面距离，机械臂下降
8. 触发吸盘
9. 机械臂回收

### 阶段 D：放置

1. 机器人导航到对应归位区
2. 根据归位区编号执行放置
3. 若归位区位置固定，可直接使用预设放置位
4. 若需要更稳，可增加末端相机放置校正

------

## 3.4 关键设计决策

### 决策 1：箱体位置由雷达地图提供，不由视觉全场搜索

**选择理由：**

- 规则明确存放区位置固定，箱体只是在存放区随机分布。
- 你们已有激光雷达全场定位
- 这样能大幅降低视觉任务复杂度，提高鲁棒性

### 决策 2：箱体识别采用“固定候选位分类”，而不是“全图检测+跟踪”

**选择理由：**

- 候选位只有 8 个，完全没必要做复杂 MOT
- 算法更简单，调试更快，比赛更稳

### 决策 3：题目识别与箱体识别分离为两条模型链路

**选择理由：**

- 题目识别属于 OCR/表达式解析
- 箱体识别属于目标类别识别
- 两者数据分布不同，合一会互相拖累

### 决策 4：抓取阶段使用末端视觉伺服，而不是仅靠全局定位

**选择理由：**

- 吸盘抓取对顶面中心误差非常敏感
- 机械臂、底盘、雷达坐标都会有累计误差
- 末端相机闭环修正是必须的

### 决策 5：归位区位置优先用地图写死，颜色识别仅作为辅助手段

**选择理由：**

- 规则说颜色可能调整，不能只靠颜色。
- 归位区编号和坐标固定，直接用地图更稳

## 4.1 题目识别接口

### Topic：`/vision/math/result`

**类型：** `task_msgs/MathResult`

```yaml
string expression_text
int32 answer
int32 mod4_zone_id
float32 confidence
bool success
int32 error_code
string error_msg
builtin_interfaces/Time stamp
```

### 错误码定义

- `0`：成功
- `1001`：未检测到题目区域
- `1002`：OCR失败
- `1003`：表达式非法
- `1004`：计算异常
- `1005`：超时


#### Response

```yaml
bool success
task_msgs/BoxSlotResult[] results
int32 error_code
string error_msg
```

### `task_msgs/BoxSlotResult`


## 4.4 抓取视觉伺服接口

### Topic：`/vision/grasp/offset`

**类型：** `task_msgs/GraspOffset`

```yaml
float32 dx_pixel
float32 dy_pixel
float32 dx_mm
float32 dy_mm
float32 distance_to_top_mm
float32 target_size_pixel
bool centered
bool distance_valid
float32 confidence
int32 error_code
string error_msg
builtin_interfaces/Time stamp
```

说明：

- `dx, dy` 表示箱体顶面中心相对图像中心的偏差
- `distance_to_top_mm` 为当前末端到箱体顶面的估计距离

### 错误码

- `0`：成功
- `4001`：未检测到顶面
- `4002`：轮廓不稳定
- `4003`：距离估计失败
- `4004`：目标超出安全区域

## 6.2 箱体识别算法建议

#### 输出

- `slot_id -> box_class_id`

#### 推荐训练方式

可以有两种：

- 类别：food/tool/instrument/medicine

## 6.3 题目识别算法建议

### 主流程

1. 采集题目图像
2. 定位显示区域
3. 透视矫正
4. 灰度/二值化/增强
5. OCR 识别表达式
6. 表达式合法性检查
7. 求值
8. `mod 4`
9. 发布高分区编号

### 为什么必须加规则校验

因为你们识别的是**四则运算题**，不是自由文本，这意味着可利用强约束：

- 只允许数字、括号、`+ - × ÷ * /`
- 括号必须匹配
- 结果必须为整数
- 字符串长度和结构要合法

这套后处理能把比赛鲁棒性拉高很多。

## 6.4 末端抓取视觉算法建议

### 目标

检测箱体顶面中心是否与图像中心重合，并估计相机到顶面的距离。

### 方法建议

分两部分：

#### A. 顶面中心检测

推荐优先级：

1. **检测顶面标识区域外接四边形**
2. 计算其中心点
3. 与图像中心比较偏差

原因：

- 顶面是近似正方形，规则尺寸固定 250mm×250mm。
- 摄像头垂直朝下时，目标几何特征清晰

#### B. 距离估计

推荐两种实现：

**首选：深度相机 / ToF**

- 直接读取顶面距离
- 开发成本最低
- 比赛更稳

**备选：单目几何估计**

- 已知顶面实际尺寸 250mm
- 通过像素边长 + 相机内参估算距离
- 需严格标定

如果你们硬件还没定，我明确建议：
**末端相机优先上深度相机。**
因为“顶面距离估计”这件事用单目能做，但现场调起来会很烦，尤其一旦光照、角度、吸盘遮挡有变化，就容易漂。
